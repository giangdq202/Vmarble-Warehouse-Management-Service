# Backend Instincts and Lessons Learned

This file serves as the Dynamic Memory for the AI Agent. It records patterns, pitfalls, and instincts discovered during development to prevent regression and repeat mistakes in the Go modular monolith.

## Core Instincts

- [Concurrency] Always use SELECT FOR UPDATE inside a transaction when performing read-then-write operations to prevent race conditions.
- [Architecture] Strictly respect module boundaries. Use deps.go for cross-module communication; never import internal/module/A into internal/module/B directly.
- [Database] Migrations must be additive and idempotent. Verify both Up and Down paths before committing.
- [Error Handling] Always return domain.NewBizError for business rule violations to ensure correct HTTP status mapping (422, 409).
- [Area Conservation] For any cutting operation, always verify that used_area + remnant_area <= source_area (BR-K03).

## Session Lessons

(New lessons will be appended here at the end of every task by the AI Agent)

- [SQL Guards] When a store UPDATE uses a WHERE guard (e.g. `AND assigned_to IS NULL`), ensure it encodes the **business invariant**, not an implementation assumption. `AND assigned_to IS NULL` silently blocks reassignment; `AND status = 'PLANNED'` correctly expresses mutability. When `RowsAffected() == 0`, always audit whether the WHERE clause is too tight before assuming a real conflict.

- [New Module Wiring] When wiring a new module in `main.go`, `cmd/server` is gitignored (pattern `server` matches). Use `git add -f cmd/server/main.go` to force-stage it. Consider adding `!cmd/server/` to .gitignore to prevent this confusion.

- [Cross-module ReceiveStock] When purchasing drives inventory creation, the adapter returns only `lot.ID` (not sheet IDs) because `inventory.ReceiveStock` does not expose individual sheet IDs. Design PO item → lot relationship, not PO item → individual sheet. Individual sheets are queryable later via `inventory_lots` join.

- [Cross-module Late-binding Cycle] When module A needs to check module B, but B also depends on A (A→B→A cycle), use a late-binding adapter with an empty `svc` field (like `woAdvanceAdapter`). Wire `adapter.svc = bSvc` after both services are constructed. Guard the adapter's method with `if a.svc == nil { return safeDefault }` to avoid nil panics during startup ordering edge cases.

- [CostingType Enum] When a record can represent two semantically different states (ESTIMATED vs ACTUAL), model this as a `type CostingType string` enum in `iface.go`, not as a boolean flag. This allows exhaustive switch in the future and makes the API self-documenting.

- [Dashboard Read-only Aggregation] For read-only dashboard endpoints that aggregate across tables, place the SQL directly in `dashboard/pgstore.go` — no migration needed, no transaction required. Use `COUNT(*) FILTER (WHERE status = 'X')` for multi-status aggregation in a single pass instead of multiple queries.

- [Cross-module Input DTOs] When a cross-module call needs to grow new fields (e.g. `BypassOverflow`, `ActorID`, `Reason` for an admin override), do NOT keep adding positional args to the interface. Migrate the dependency interface to take a struct (`PreAssignSheetInput` in inventory, `PreAssignSheetRequest` in production's `deps.go`) and let the adapter in `cmd/server/main.go` translate between the two shapes. This keeps each module's public DTO local and lets either side add fields without rewriting the other.

- [Defense in depth for admin overrides] When a route guard restricts a privileged flag (e.g. `bypass_overflow` requires `RoleAdmin`), validate the same invariant again in the consuming service layer. Even if the route is mis-wired, the inner module rejects the call. For audit-bearing actions, also assert that `ActorID` and `Reason` are populated at the inventory boundary — a missing actor on a bypass row is worse than a 400.

- [Best-effort advisory paths] Some side effects are advisory only — e.g. writing a `REMNANT_BYPASSED` audit row when a planner ignores remnant suggestions. The primary operation (CreateWorkOrder) must succeed even if the advisor errors. Run the advisory call AFTER the primary write commits, swallow advisor errors with `slog.Warn(... "wo_id", wo.ID, "err", err)`, and never roll back. The audit miss is recoverable; failing to create the work order over a logging hiccup is not. Tests must cover both success and advisor-error branches to prove the API stays 2xx.

- [Audit log as multi-entity] An `inventory_audit_log` that originally logged remnant rows (`entity_type = 'REMNANT'`) eventually needs to log work-order events (`entity_type = 'WORK_ORDER'`) and structured payloads (`metadata jsonb`). Keep `entity_type` as a free-form string with a service-layer allow-list, and add `metadata jsonb` early — a single `reason text` column quickly becomes overloaded once you need to store an array (e.g. `suggested_remnant_ids`). Index `(action)` separately from `(entity_id)` because review queries fetch by action across many entities.

- [Re-scope before code] An issue title like "API X trống" can hide a totally different bug — before writing code, read the FE route and the BE handler that backs it. `/report-cut` was filed as "missing report data" but turned out to be a kiosk form mis-labelled "Báo cáo" in bottom-nav. Had I jumped straight to `ComputeCost`-style code, the "fix" would not have solved the user's actual pain. Spend 5 minutes verifying the cause before picking up the DoD verbatim.

- [Typed-nil params for optional SQL filters] For pgx queries with optional filters (`$1::uuid IS NULL OR col = $1`), declare the placeholder as `var userID any = nil` and reassign the concrete `uuid.UUID` / `time.Time` when set. Never pass a `uuid.UUID{}` zero-value — pgx will bind it as an all-zeros UUID and your `IS NULL` branch never fires. This pattern scales cleanly for 3–4 optional filters without resorting to dynamic SQL building.

- [Guard placement with BR precedence] When adding a new compute-time guard (zero-cost, invalid state) to a service method that already has BR guards (finalized immutability, transition checks), **place the new guard AFTER the higher-precedence BR checks**. Otherwise a "nicer" VN error message leaks and masks the real violation. In `ComputeCost`: Finalized check must fire before the zero-cost guard so BR-C04 wins when both conditions hold. Add an explicit test `..._ZeroCost_ReturnsAlreadyFinalized` to lock this ordering.

- [Test migration strategy when tightening a guard] When a new guard fails on pre-existing tests that were coincidentally zero-cost COMPLETED fixtures, do NOT dilute the guard with exceptions. Instead, identify which tests are testing the guarded condition vs. testing unrelated subjects (insert path, update path, error propagation) — migrate the unrelated ones to use `plannedWO` (exempt from the guard) so they keep focusing on their real subject. Only the handful of ACTUAL-specific tests need to be updated with real cost data.

- [Proxy attribution when schema lags] For reports that need "who did X" but the source table lacks a `created_by` column (like `cutting_records`), use the existing FK to another table that holds assignment data (`work_orders.assigned_to`) as a proxy. Document the proxy clearly in the DTO doc comment + issue body so future readers know the semantic limitation: reassignment will rewrite history. File a follow-up issue for the proper migration if precision matters.

- [Polling vs SSE discovery] Backend already had Postgres LISTEN/NOTIFY + SSE endpoint `/notifications/stream` with broker fan-out, but FE never connected — so user-facing "realtime" was pure 30-60s polling. When the user complains "update chậm," grep FE for `EventSource` / `notifications/stream` / `refetchInterval` before assuming the pipe is missing. The fix split naturally into BE (expand event types + topic/roles broadcast) and FE (wire consumer + invalidate queries). Native EventSource cannot send Bearer headers — use `@microsoft/fetch-event-source` or token via URL query.

- [Tighten BE contract before FE catches up = breakage] Before adding a new required field to an existing input DTO, grep the FE for the new field name AND for the old payload shape. If FE has not adopted yet, the BE merge will 400 a live flow. Two valid responses: (a) ship FE companion issue + link block on BE PR before merge, or (b) make the field optional with a deprecation window, log warnings on the implicit branch, then tighten in the next sprint. Never assume "FE will catch up" — file the FE issue at the moment you draft the BE PR, not after.

- [Bulk fixture migration via brace-walking script] When a single required field is added to a struct used in 30+ test fixtures, a Python script that walks `{` / `}` depth and inspects body content is far faster than manual edits — but a naïve regex append leaves `,,` artifacts where the existing field already had a trailing comma. Always run a `sed -i '' 's/,,$/,/'` cleanup pass and `go vet` before assuming the migration is clean. Repeated regex substitutions on Go syntax are surprisingly easy to corrupt.

- [Scaffolding-first audit before designing] When picking up an issue with a long DoD checklist (#250 listed migration, two endpoints, response enrichment, audit, tests, swagger), do NOT design from scratch. First grep for the table name, the route pattern, and the expected service method names — most of the scaffolding usually already exists from earlier sprints. For #250, the table, store methods, `CreateAdjustment` / `ListAdjustments`, and both POST/GET routes were already shipped; the only real gap was the response shape (`adjustments[]` + `effective_total`). Skipping this audit would have produced a redundant migration and duplicate handler.

- [Composite read DTOs over enriched gets] When a DoD asks "GET response should include child collection X and computed field Y", resist the temptation to bloat the existing primary GET. Add a sibling `/detail` route returning a composite DTO (`CostingRecordDetail` = record + adjustments + effective_*) so list rows / drilldowns that only need the flat record stay cheap, and the dialog gets exactly the shape it needs. The flat get becomes the "is this record finalized?" probe; the detail get is the "render the dialog" call. Two purposes, two payloads.

- [Immutability assertion in folding tests] When a method folds child rows into parent-derived "effective" values, the test must assert that the parent fields stay untouched after the fold — not just that effective totals are correct. `TestGetCostingRecordDetail_WithAdjustments_FoldsDeltas` checks both `EffectiveMaterial == 105_000` AND `Record.MaterialCost == 100_000` (BR-C04). Without the immutability check, a future refactor that accidentally mutates `record.MaterialCost = record.MaterialCost.Add(a.DeltaMaterial)` would still pass the totals check but silently break the immutability invariant.
- [Optional cross-module deps via paired-nil guard] When adding new deps (e.g. `StockChecker` + `BOMReader` for BR-K01) to a service constructor that 30+ test fixtures already pass `nil` for related deps, do NOT migrate every existing constructor call. Add a third constructor (`NewServiceFull`) that takes the new deps; keep the old `NewService` / `NewServiceWithRemnantAdvisor` for tests. In `service.go` guard the new code path with `if svc.stk != nil && svc.br != nil { ... }` — both deps must be wired together since one without the other is meaningless. Production gets the full guard via `main.go`; tests opt in only when the test exercises the guarded path.

- [BOM unit semantics ambiguity → floor at 1] When BOM `QuantityPerUnit` could mean sheets, area, or ratio depending on the material, do NOT multiply blindly into a stock requirement. Pick the most conservative interpretation that still catches the headline bug — for #248 that was floor at 1 sheet per unit (catches "plan 50, kho 10" while never producing a false 422 from over-allocation). Document the simplification in the doc comment + PR body and file a follow-up. False rejects on a P0 path are worse than a too-loose guard.

- [Existing branch carries unrelated commits] When `git checkout -b feat/B` is run on top of branch `feat/A` that hasn't merged yet, the new branch ALSO carries A's commits — `git log --oneline dev..HEAD` reveals it. Before pushing the B PR, recreate the branch off `dev` (`git branch -D feat/B && git checkout dev && git checkout -b feat/B`) instead of force-resetting. The reset is destructive on shared branches; the recreate is a no-op for unstaged work and surfaces orphaned commits explicitly.

- [SSE schema lives on the handler doc] When the FE has to render different toasts/queries per event type, the canonical source-of-truth must be a comment on the SSE route handler — not a separate doc that drifts. Document every emitted `type` value, who routes by `UserID` vs `Roles`, and the exact JSON shape on the `// @Description` of `/notifications/stream`. The next FE engineer doing query-invalidation reads the handler comment first; anything missing there will be reinvented. For #258 we listed all 5 event types + each routing audience inline so FE #194 could wire `useEffect` invalidations without reading any backend code.

- [Per-subscription dedup beats per-channel dedup] When fan-out routes by both `UserID` (assignee personal) AND `Roles` (manager broadcast), an assignee who is also in the manager role would receive duplicates if the broker iterates the two paths separately. Refactor `clients map[string]map[int64]chan` → `clients map[int64]*subscription{userID, role, ch}` and use OR-match per subscription. Each connection gets at most one event regardless of how many routing rules match. Add a regression test (`TestBroker_PersonalAndRoleBoth_DeduplicatesPerSubscription`) the first time you wire role broadcasts because the bug only surfaces when one user holds two routing keys.

- [Best-effort notify wiring pattern] Every new SSE publish call site (4 added in #258) follows the same recipe: (a) define the deps interface in the consuming module's `deps.go` with a clear "best-effort" doc-comment, (b) wire the publisher as the implementation in `main.go`, (c) call `svc.notifier.NotifyXxx(...)` AFTER the business write commits, (d) wrap the error with `slog.Warn(... "wo_id", woID, "err", err)`, NEVER return it. Tests must cover the happy-path-publishes-once, notifier-error-doesnt-fail-request, and validation-failure-skips-publish branches per call site. Copy-paste the test names so the next reviewer instantly recognizes the contract.

- [Variadic-with-type-switch deps for backward compat] When adding optional deps to a constructor with 20+ existing call sites (`barcode.NewService(st, ...any)`), prefer extending the variadic `deps ...any` + type-switch dispatch over adding a `NewServiceFull`. Existing tests pass `nil` or skip the dep entirely; new wiring just appends `notifier` to the variadic call. The type-switch already handles `WorkOrderGateway` and `UserLookup` — adding `case ScanNotifier:` is a one-liner. Keeps the diff minimal and avoids the "two constructors" footgun where future devs pick the wrong one.

- [Reports as adapter-only module] An export module should own NO domain entities and NO list APIs. Define row types + reader interfaces in `iface.go`/`deps.go`; then in `main.go` either (a) project pgxpool queries straight into the row types, or (b) delegate to an existing service method (e.g. `costing.ListWasteReport`) for aggregations the source module already does. This keeps the "what data?" decision in the source module and the "how to format?" decision in reports — two responsibilities, two locations. The `reports` module never imports `costing` / `catalog` / etc.

- [Period filter: To-exclusive end-of-day] When the FE asks "give me records from May 1 to May 15", they almost always mean inclusive-end. SQL `created_at < $2` requires May 16 as the bound. Resolve this in the handler: if `?to=YYYY-MM-DD` is set, `end := t.Add(24 * time.Hour)` before passing to the service. Document the exclusivity choice in the handler comment so future readers don't "fix" the bug by changing `<` to `<=`. Equal `from=to=2026-05-15` returns May 15 records, not zero rows.

- [Excel formatting via per-column NumFmt registry] excelize has no "format this column with thousand separator" — you have to pre-compile each format into a styleID and apply per-cell. Build the styleID once outside the row loop, store in `colStyles[colIdx]`, and apply with `f.SetCellStyle(sheet, cell, cell, styleID)` only on cells that need it. For VND amounts use `"#,##0\" đ\""` so the cell stays a number (sortable, sum-able) but renders with the đ suffix. Date cells should be plain strings already in dd/mm/yyyy — Excel auto-coerces to date format on locale and the round-trip is lossy.

- [Workbook header pane freeze + bold] Single line per workbook: `f.SetPanes(sheet, &excelize.Panes{Freeze: true, YSplit: 1, TopLeftCell: "A2", ActivePane: "bottomLeft"})`. Combined with a bold header style, accountants get a usable spreadsheet without touching Excel. Cost: one extra style + one API call per workbook. Don't skip — the difference between "workbook-shaped output" and "thing accountant actually opens twice" is the freeze pane.

- [Branch hygiene after PR merge] After PR #N merges, before cutting branch for #N+1: `git checkout dev && git pull --ff-only origin dev` — never `git checkout -b feat/N+1 feat/N` because (a) feat/N may already be deleted upstream and (b) feat/N+1 inherits feat/N's history including the merge commit. The right pattern is always: PR #N merges → switch to dev → fast-forward → branch off dev. Re-running the same `git checkout -b` from a stale branch is the #1 source of "PR has 30 commits when I only made 5".

- [Cascade cancel = cross-module verb, not state-machine extension] When planning needs to cancel APPROVED plans and tear down PLANNED work orders (#249), the temptation is to extend the WO state machine to allow `PLANNED → CANCELED`. Resist it: WO transitions stay strictly forward, and the new terminal state is reached via a *different verb* (`CancelPlannedByPlan`) that runs a single SQL UPDATE with a `WHERE status = 'PLANNED'` guard. The `AdvanceStatus` state machine (BR-P02) explicitly refuses to enter or leave CANCELED — that lane is owned by the cascade alone, called only from `planning.CancelPlan` after upstream validation. Two paths into the same row, one strictly forward, one strictly via cascade. Never mix them.

- [Audit-trail UPDATE atomic with status flip] When adding `canceled_reason` / `canceled_at` / `canceled_by` columns next to an existing `status` column (#249), do NOT wire it as a status flip + a follow-up UPDATE for the audit fields. A single `UPDATE … SET status='CANCELED', canceled_reason=$1, canceled_by=$2, canceled_at=$3 WHERE id=$4` keeps the audit trail in lockstep with the state change — no half-transitioned row can exist if the second statement crashes. The DRAFT path can still call the plain `updatePlanStatus` because there is no audit data to record; the APPROVED path must use the new combined method.

- [Two constructors for optional deps, not nil-guard inside ctor] When adding `WorkOrderCanceller` to `planning.Service` (only used by APPROVED cancels), keep the original `NewService(store)` and add `NewServiceWithDeps(store, woCanceller)`. Tests that exercise DRAFT-only flows pass through `NewService` and never need a stub canceller; production wires the full constructor in `main.go`. Inside the service, guard with `if svc.woCanceler == nil { return ErrPreconditionFailed }` only on the branch that actually needs it. This is the same pattern as `NewServiceFull` from #248 — both deps optional at the type level, mandatory at the branch level.

- [Cross-module audit via pgxpool adapter, not inventory.Service dependency] When `costing.CreateAdjustment` needs to write a row to `inventory_audit_log` (#250), do NOT add `inventory.Service` as a dep — the audit log is a shared cross-module ledger keyed by `entity_type`, not an inventory entity. Define a thin `AuditLogger` interface in `costing/deps.go` with module-scoped DTO (`AuditCostingAdjustmentInput`), and implement it in `cmd/server/main.go` as `costingAuditAdapter` that runs `INSERT INTO inventory_audit_log` directly via pgxpool. Costing stays free of the inventory module; the audit table semantics (gen_random_uuid, NOW(), JSON metadata) are owned by the adapter, not the service. Same pattern works for any future module that needs to drop a row into the shared ledger without inheriting an entire SDK.

- [Streaming Excel export via callback iterator + StreamWriter] The original reports module loaded the full result set into `[]Row`, then duplicated it into `[][]any`, then handed the lot to excelize's in-memory `SetCellValue` — three full copies in RAM at once, ~1.5GB peak for 1.5M rows. Refactor (#273) the reader interfaces from `List*(...) ([]Row, error)` to `Iterate*(..., yield func(Row) error) error`, and the service signature from `(...) (ExportFile, error)` (returns bytes) to `(... w io.Writer, ...) (filename string, err error)`. Inside the service, drive `excelize.NewStreamWriter` with `sw.SetRow(axis, []any{excelize.Cell{StyleID, Value}, ...})` per row — heap usage drops to O(1) per row regardless of dataset size. The handler must compute filename + write `Content-Disposition` BEFORE calling the export, because once the StreamWriter flushes, the response is mid-flight and you cannot rewrite headers. Smoke-test with 10k rows in CI to prove the path doesn't accidentally re-introduce buffering.

- [Bound period exports at the service entry, not at the SQL layer] When an Excel export accepts `?from=&to=`, validate the span (`MaxPeriodDays = 90`) inside `Period.validate()` BEFORE opening any pgx rows or excelize file. The cap is a UX guard — accountants who type a year-long range get a 4xx with a Vietnamese error, not a multi-minute streaming download that may still OOM the StreamWriter's xml temp file. Open-ended exports (only one bound set) are permitted because the streaming pipeline keeps RAM bounded regardless of result count. Catalog-style endpoints with no time bound (e.g. SKUs) cap via SQL `LIMIT $1` bound to `MaxSKURows = 50000` instead — same goal, different lever.

- [Dual-root via XOR CHECK, not nullable columns alone] When pivoting a foundational table from "always rooted by X" to "rooted by either X or Y" (#289 production_plans po_id → po_id|sales_order_id), do NOT just drop the NOT NULL on the old column and add the new one. Add a `chk_*_root` CHECK constraint that requires exactly-one-of via `((a IS NULL) <> (b IS NULL))`, and refactor every read path to LEFT JOIN both parent tables with `*string` scan targets. The DB constraint becomes the contract — service code only validates input shape (`(POID==nil) == (SOID==nil)` returns ErrInvalidInput). Without the CHECK, a future caller can write rows with both roots set or neither, and the read joins silently return wrong data.

- [Cross-module composite operation = N transactions, document the seams] Sales `SplitToPlan` has to compose planning.CreatePlan + production.CreateWorkOrder + a sales-side qty_planned bump. Each module owns its own pool/tx, so you cannot wrap the whole thing in one pool-level transaction without restructuring every module to accept an injected `pgx.Tx`. Accept the seams: Phase 1 (`withTx` lock SO+lines FOR UPDATE, validate qty caps), Phase 2 (cross-module call — own tx, no rollback hook), Phase 3 (`withTx` bump qty_planned + auto-flip status). Document the failure window in `deps.go` so the next reader knows the contract. Surface Phase-3 failures as `ErrPreconditionFailed` with the orphan plan code in the error message — operations team needs to know what to reconcile.

- [Daily counter via INSERT … ON CONFLICT … RETURNING] When generating sequence codes scoped per-day (`SO20260525-001`), don't write SELECT-then-UPDATE — it's racy under concurrent inserts even with `FOR UPDATE`. Create a tiny counter table keyed by date and run `INSERT INTO sales_order_code_counters (day, last_seq) VALUES ($1, 1) ON CONFLICT (day) DO UPDATE SET last_seq = sales_order_code_counters.last_seq + 1 RETURNING last_seq`. Atomic in one statement, race-free, no advisory lock needed. Same pattern works for any per-bucket monotonic counter (per-customer invoice, per-month batch, etc.).

- [Validation timing: save vs commit] BR-S05 (export orders need incoterm/port_of_loading/port_of_discharge) does NOT run in CreateSO or PatchSO — only in ConfirmSO. Reason: the accountant typing in fields one-by-one shouldn't get rejected on every save just because they haven't filled in the port yet. DRAFT is a sandbox; CONFIRM is the gate. Mirror this for any "validation should run on commit, not on save" flow — file the validation under the state-transition method, not the data-mutation method. The handler signals which check is which by returning DRAFT 200s freely and CONFIRM 400s with `ErrInvalidInput`.

- [twoShotStore harness for multi-tx services] Service methods that call `withTx` multiple times (e.g., SplitToPlan Phase 1 + Phase 3) benefit from a tiny test harness: wrap mockStore with a `twoShotStore{first, second *mockTxStore, calls *int}` that hands back different `mockTxStore` instances based on a call counter. Avoids stateful boolean toggling inside a single mock and lets the test name read naturally — `TestSplitToPlan_QtyPlannedTxFails_FlagsOrphan` immediately conveys "Phase 1 ok, Phase 3 fails". Same pattern applies any time a service calls the same store method twice with different expected outcomes.

- [Cross-module Tx leak as the documented exception, not the rule] Most cross-module composite ops (sales SplitToPlan) accept N transactions and document the seam. Delivery `Seal` is the exception: the container.status flip + qty_shipped bump on every underlying SO line MUST be one DB transaction or the warehouse can ship goods that the SO ledger never recorded as shipped. Solve it by exposing `RecordShipmentTx(ctx, tx pgx.Tx, items)` on the consumed module (sales) — caller hands its own tx in and the recorder runs INSIDE it. Document the leak loudly in `deps.go` of the calling module ("deliberate exception to modules-own-their-pool"). Implement it as a pgx-typed interface so the tx never gets wrapped in a custom abstraction the caller would have to satisfy. Use sparingly — N transactions is the default; one tx across modules is the audited deviation.

- [withTx hands raw + wrapped tx so cross-module hooks can join] When the service runs cross-module hooks inside its own transaction, the `withTx` helper must yield BOTH the txStore wrapper (for typed module-internal ops) AND the raw `pgx.Tx` (for cross-module hooks that take `pgx.Tx`). Signature: `withTx(ctx, fn func(tx txStore, raw pgx.Tx) error) error`. The wrapper holds the raw tx internally and exposes it through the second arg. Saves you from leaking `pgx.Tx` into the typed wrapper or — worse — running the hook on a fresh connection that lives outside the wrapping tx.

- [Container capacity overhead via env var, not a magic constant] Real-world packing rarely fills an ISO container to its theoretical max — operators routinely shave 3–7% headroom for irregular shapes. Solve BR-D02/D03 by reading `CONTAINER_CBM_OVERHEAD_PCT` (default 5) from config and computing the cap as `MaxCBM * (1 + overhead/100)`. Default the overhead in `config.Config` so the service constructor takes a single float; tests pass 0 to disable, prod gets the env-driven 5. Cuts the "AddLine rejected at 99.6% fill but the truck is half-full" support tickets in half. Document the overhead in the module README so frontend doesn't show a misleading "0.4% remaining" while the API still accepts another snapshot.

- [Deterministic lock ordering for symmetric pair operations] TransferLine moves a line between two containers; two foremen working in opposite directions on the same pair of containers can deadlock if each grabs `source` first. Sort the two UUIDs lexicographically (`bytes.Compare(a[:], b[:])`) and lock in that order regardless of which is "source" and which is "target", then resolve back to src/target inside the tx. Same pattern for any pair-symmetric op (account-to-account transfer, swap, etc.) — the canonical order is whatever total order is stable across processes; UUID byte-compare is the cheapest option.

- [Snapshot-supplied physical fields when catalog isn't ready] Container lines need CBM + weight to enforce capacity, but the catalog SKU table doesn't carry `height_mm` / `weight_kg` / `cbm` yet (#294 will add them). Don't block the delivery module on that schema work — accept `cbm_total` + `weight_kg_total` from the client at AddLine time and snapshot them on the row. When #294 lands, the FE simply switches from "look up sku.cbm and multiply" to passing the same numeric through; no API change required. Document the eventual migration path in the input DTO doc-comment so future readers know the snapshot is temporary.

- [State-aware mockTxStore for stateful aggregate reads] When AddLine validates capacity by summing existing lines, the test harness must thread state across calls — a stub `sumLinesAggregates` returning constants can't catch "first line passes, second pushes over the cap". Build the mock around `containersByID map[uuid]Container` + `linesByID map[uuid]ContainerLine` and compute aggregates by iterating `linesByID` filtered by container. The mock then mirrors real DB semantics: insert a line and the next sum reflects it. Pays for itself the first time you write a "two adds, second one overflows" test.

- [Setter-injected hook breaks two-way module cycle] When module A consumes module B (packing -> production WorkOrderGateway) AND module B needs to fire a hook into A on a state transition (production -> packing FG hook on COMPLETED), constructor wiring deadlocks: each module needs the other to exist first. Solution: keep B's constructor unaware of A, expose a setter on the concrete type (`SetFinishedGoodsHook`), and in `main.go` attach the hook AFTER both services exist via type assertion: `if h, ok := bSvc.(interface{ SetFinishedGoodsHook(...) }); ok { h.SetFinishedGoodsHook(...) }`. The interface assertion keeps B's `Service` interface clean (setter doesn't pollute the public contract) while still letting tests opt out by leaving the hook nil. Same trick for delivery -> packing FG track.

- [Best-effort hook fires AFTER the business write commits] When wiring a cross-module hook on a state transition (production -> packing FG creation on COMPLETED), call the hook AFTER `s.updateWorkOrderStatus` returns success — never inside the same transaction. Reason: the hook may take seconds (barcode generation, HTTP, etc.) and a hook failure that rolls the WO transition back creates a worse outcome than an orphan hook (warehouse can backfill via the idempotent CreateFromCompletedWO call when the broker recovers). Wrap the hook call with `slog.Warn` + swallow; the transition is already durable. If the hook ever DOES need atomicity, that's a different problem — solve it via the cross-module Tx leak pattern (sales.RecordShipmentTx), not by stuffing async work into the critical path.

- [Idempotent state-transition hook = freedom to retry] CreateFromCompletedWO returns existing rows when `selectFGByWorkOrderID(woID)` already has them — no error, no duplicate barcodes. This buys two operational properties: (a) the production hook can fire-and-forget without coordination, (b) a manual "rebuild FG pool for WO X" admin tool can call the same method without special-casing. The DB-level UNIQUE on `barcodes` is the safety net, but service-level idempotency keeps the call cheap and observable (the kiosk doesn't see "barcode already exists" 409s mid-flow). Always design state-transition hooks this way: same input, same output, no side-effect difference between first call and Nth call.

- [SKIP LOCKED for soft-allocation pools] When delivery.AddLine reserves N FG rows from packing's pool, two concurrent AddLine flows targeting the same SO line must NOT block each other — the pool is soft, callers are happy with whatever they can get. Use `SELECT … WHERE status='AVAILABLE' AND sku_id=$1 LIMIT $2 FOR UPDATE OF fp SKIP LOCKED`. Each tx grabs a disjoint slice of the available rows, neither stalls. Returning fewer rows than requested is the contract — caller decides whether the shortfall is fatal (delivery.AddLine just logs it; a stricter caller could 422). Without SKIP LOCKED, two foremen scanning the same SKU at the same kiosk would queue serially even though the pool has 50 free units.

- [Manage cross-module wiring depth via post-construction setters, not deeper deps] When wiring grew to 3+ modules with bidirectional needs (production <-> packing <-> delivery), the temptation was to bake everything into constructor params and end up with 11-arg `NewService` calls. Resist. Each module's constructor takes only its UPSTREAM deps (the things it must call). Hooks pointing DOWNSTREAM (delivery.SetFGTracker, production.SetFinishedGoodsHook) get installed via setter after both services exist. Result: each constructor signature stays under 6 args, each setter is single-purpose, the wiring order in main.go reads top-to-bottom without cycles. The interface-assertion idiom (`if h, ok := svc.(interface{ Setter(...) }); ok`) keeps the test surface minimal — tests that don't need the hook leave it nil and the interface stays unbloated.

- [Schema-level invariants for status + reference pairs] When a row's status implies whether a foreign-key column should be set (RESERVED/LOADED FG must reference a container_line; AVAILABLE must NOT), enforce it with a CHECK constraint paired against status: `CHECK ((status IN ('RESERVED','LOADED') AND container_line_id IS NOT NULL) OR (status NOT IN ('RESERVED','LOADED') AND container_line_id IS NULL))`. Without it, a future bug that flips status without updating the FK (or vice versa) silently persists, and the inconsistency only shows up when a downstream report tries to JOIN. The CHECK is one line — it makes "impossible state" actually impossible.

- [Guard double-resolve via SQL WHERE on the resolution column itself] When ResolveDefect must reject a second resolve attempt, a service-layer "fetch defect, check resolution IS NULL, then UPDATE" has a race window. Push the guard into SQL: `UPDATE fg_defect SET resolution=$1, ... WHERE id=$2 AND resolution IS NULL`, then check `RowsAffected==0` and return ErrInvalidTransition. Atomic, race-free, no advisory lock. Same idiom for any "single-shot" mutation (one-time approve, one-time void, etc.). The CHECK constraint `(resolution IS NULL AND resolved_by IS NULL) OR (resolution IS NOT NULL AND resolved_by IS NOT NULL)` is the second safety net.

- [Multi-edge state machine: map[X][]X over map[X]X] When a state has multiple valid next states (#292 IN_PROCESSING → COMPLETED **or** PARTIAL_COMPLETE), do not work around it with two booleans or a separate "branch state". Migrate `woTransitions` from `map[X]X` to `map[X][]X` and rewrite `CanTransitionTo` as membership check, `Next()` as `nexts[0]` for the canonical happy-path successor. Existing single-edge entries become slices of length 1 with no semantic change. The first-element-is-canonical convention keeps the dashboard "what's next?" hint deterministic without introducing a new annotation.

- [Service-level pre-check + store-level locked re-check] For BR-P05-style "must be in state X" guards on a write that races against itself, do BOTH: (a) fast service-level pre-read of the WO + status check, gives callers a 409 in the common case without holding a row lock; (b) authoritative `SELECT … FOR UPDATE` re-check inside the store's atomic op, gives correctness when two concurrent calls slip past the pre-check. The pre-check is for UX (sub-ms 409 in the unhappy path); the FOR UPDATE re-check is for safety. Returning the same `ErrInvalidTransition` from both keeps the error surface consistent — the caller cannot tell which branch fired.

- [Carry-over WOs preserve qty_planned, not increment it] When PartialComplete spawns a carry-over WO for the shortfall, the carry-over inherits `sales_order_line_id` so SO traceability is preserved — but `sales_order_lines.qty_planned` is **not** incremented. Production owes more units against the same SO commitment; the SO scope did not grow. Reverse: if the carry-over later partial-completes again, `qty_planned` still does not move. The chain length is bounded by the original SO commitment. Document this in the BR (BR-P08) so future schema reviewers don't add a "denormalized qty_outstanding" column thinking it's tracking something other than `original.qty - sum(actual_qty across chain)`.

- [Single-purpose enum CHECK over generic TEXT for hand-edited columns] `shortfall_reason` could have been free-form TEXT with service-level validation, but the four-value enum (`MATERIAL_SHORTAGE | DEFECT | TIME_SHORTAGE | OTHER`) is small and stable. Pin it as a DB CHECK so a misbehaving import script or a future `psql` REPL session cannot insert junk reasons. The CHECK is the second line of defence; the service still validates first to give a 400 instead of a constraint violation. When the enum needs a fifth value, add a migration — the service+CHECK pair is one search-and-replace and survives DB-level pokes.

- [Migrate optional store args to a Filter struct before adding the third one] When a store method's optional parameter list grows past two (#313 added sku_id/from/to/search alongside the existing finalized flag on `selectCostingRecordsKeyset`), do NOT keep extending positional pointer args (`*bool, *uuid.UUID, *time.Time, *time.Time, string`). Migrate the signature to take a `XxxListFilter` struct in `iface.go`, pass it through `service.go` -> `store.go` -> `pgstore.go` unchanged, and update the mockStore field from `listFin *bool` to `listFilter XxxListFilter`. Single-fixture tests then write `CostingListFilter{Finalized: &b}` instead of `&b`, and adding the next filter in 6 months is one struct field, not a 4-file ripple. The existing `_pre_check_then_locked_recheck` and `cross_module_input_dtos` instincts both predicted this — the filter is just the same pattern at the read path.




- [Bulk import: validate everything before any insert, return ordered row errors] Customer SKU mapping bulk import (#304) had two competing pulls: (a) "fail-all" semantics so a malformed CSV never half-applies; (b) per-row error feedback so the FE can show "row 7: duplicate code, row 12: unknown sku". Resolved by walking the rows twice in service.go: first pass validates each row (trim+length, intra-batch dup detection via `seenCodes map[string]int`, cross-module SKU existence) accumulating `[]BulkImportRowError{Row, Code, Message}`; if any errors collected, return early with `ErrInvalidInput` and zero inserts. Only when validation is clean does a single `bulkInsertCustomerSKUMappings` tx fire. The store-level tx still rolls back atomically on a constraint violation that escapes pre-check (FK race), but in steady state the service rejects the batch before holding any row lock. Pattern applies to any future CSV/Excel import — PO, work order, defect import — same shape: validate-all → error-or-commit, never partial. The `Row` field is 1-indexed to match the Excel "row 1 = header" mental model the FE renders.

- [Composite-PK audit rows: random entity_id + real key in metadata] When a child table uses a composite primary key (customer_sku_mappings is keyed by `(customer_id, customer_sku_code)`), the shared `inventory_audit_log` schema cannot store the PK in `entity_id uuid`. Two viable shapes: (a) hash the composite to a UUID v5; (b) generate a fresh `gen_random_uuid()` for `entity_id` and put the real key in `metadata` JSON. Pick (b) — the audit row is keyed by `entity_type='CUSTOMER_SKU_MAPPING'` for queries, the metadata holds `{customer_id, customer_sku_code, sku_id, previous_sku_id?, rows_imported?}`, and accountants who need to reconstruct a change `WHERE entity_type=... AND metadata->>'customer_sku_code' = 'X'` get human-readable filtering instead of having to recompute a v5 hash. Same approach when the next composite-key entity (price_list_lines, etc.) needs auditing.

- [Best-effort audit hook is fire-and-log, never propagated] `salesMappingAuditAdapter.LogCustomerSKUMapping` returning a non-nil error must NOT roll back the mapping mutation. Service wraps each call in `logMappingAudit` that swallows the error via `slog.Warn`; the public Service interface does not propagate audit failures. This mirrors `salesShipmentAdapter` and the production FG hook — if the audit table is briefly unreachable, business writes still succeed and an ops dashboard surfaces the dropped audit row separately. The cross-module Tx leak pattern (sales.RecordShipmentTx) is the explicit deviation from this rule; audit hooks never qualify because their consistency requirement is "eventually queryable", not "atomic with the business write".
